## 一、核心问题：为什么路由以 `/webroot/decision` 开头

### 1. 路由组成分析

FineReport的完整URL路由格式为：

```
/webroot/decision/xxx
```

这个路由由两部分组成：

| 组成部分 | 来源 | 说明 |
|---------|------|------|
| `/webroot` | Tomcat部署配置 | Web应用上下文路径(Context Path) |
| `/decision` | 应用内部配置 | Servlet名称，可配置 |

### 2. `/webroot` 的来源

`/webroot` 是Web应用的上下文路径，由以下因素决定：

1. **Tomcat部署结构**：
   ```
   webapps/
   └── webroot/          # 应用目录名即为默认的Context Path
       └── WEB-INF/
   ```

2. **代码中的定义** (`ProjectConstants.java:34`)：
   ```java
   public static final String WEBAPP_NAME = "webroot";
   ```

### 3. `/decision` 的来源

`/decision` 是默认的 **Servlet名称**，决定了哪些请求由 FineReport 的核心 Servlet 处理。

#### 3.1 什么是 Servlet 名称？

在 Java Web 应用中：
- **Servlet 名称** 是一个标识符，用于在应用中唯一标识一个 Servlet
- **Servlet 映射** 基于这个名称，将 URL 模式与 Servlet 关联起来

```
URL: /webroot/decision/login
           ↓
匹配规则: /decision/*
           ↓
Servlet名称: decision
           ↓
实际处理的Servlet: FineDispatcherServlet
```

#### 3.2 核心配置类 `ServerConfig.java`

```java
// ServerConfig.java:16
private Conf<String> servletName = Holders.simple("decision");

// ServerConfig.java:50-52
public String getServletMapping() {
    return "/" + this.servletName.get() + "/*";
}
```

#### 3.2 平台基础配置接口 `PlatformScaffoldBasicConfigProvider.java`

```java
// PlatformScaffoldBasicConfigProvider.java:38-44
default String getServletName() {
    return "decision";
}

default String getServletMapping() {
    return "/" + getServletName() + "/*";
}
```

#### 3.3 Servlet注册过程 (`PlatformScaffoldServletStarter.java`)

```java
// PlatformScaffoldServletStarter.java:23-28
public static void startServlet(ServletContext servletContext, Servlet dispatcherServlet) {
    ServletRegistration.Dynamic servlet = servletContext.addServlet(
        PlatformScaffold.getBasicConfigProvider().getServletName(),  // "decision"
        dispatcherServlet
    );
    servlet.addMapping(new String[]{
        PlatformScaffold.getBasicConfigProvider().getServletMapping()  // "/decision/*"
    });
    servlet.setLoadOnStartup(1);
    servlet.setAsyncSupported(true);
}
```

### 4. 完整URL构建过程

在 `DecisionServletInitializer.java:90` 中：

```java
SateVariableManager.put(WebServerConstants.FINE_CONTEXT_PATH, servletContext.getContextPath());
SateVariableManager.put(WebServerConstants.FINE_SERVLET_URL,
    servletContext.getContextPath() + "/" + ServerConfig.getInstance().getServletName());
```

最终构建的完整Servlet URL为：`/webroot/decision`

## 二、路由架构总览

### 1. 技术架构

FineReport使用 **Spring MVC** 作为Web框架：

- **核心Servlet**: `FineDispatcherServlet` (继承自Spring的`DispatcherServlet`)
- **配置类**: `DecisionConfiguration`
- **组件扫描**: `@ComponentScan` 自动扫描Controller

### 2. 路由分发流程

```
HTTP请求
    ↓
Tomcat (根据Context Path /webroot 分发到应用)
    ↓
FineDispatcherServlet (根据Servlet Mapping /decision/* 拦截)
    ↓
拦截器链 (Interceptor Chain)
    ↓
HandlerMapping (根据@RequestMapping匹配Controller)
    ↓
Controller处理业务逻辑
```

### 3. 拦截器配置 (`DecisionConfiguration.java`)

| 拦截器 | 路径模式 | 功能 |
|-------|---------|------|
| SystemAvailableInterceptor | `/**` | 系统可用性检查 |
| MigrationInterceptor | `/**` | 迁移处理 |
| EncryptionInterceptor | `/**` | 加密处理 |
| DecisionInterceptor | `/**` | 决策核心拦截 |
| SecurityAccessInterceptor | `/view/**`, `/v10/entry/access/**` | 安全访问控制 |
| SessionCheckInterceptor | `/v10/entry/access/**` | 会话检查 |

**方式一：修改配置文件**

修改Tomcat部署配置或应用目录名即可。

## 六、总结

FineReport的路由 `/webroot/decision` 开头是由以下两个因素共同决定的：

1. **`/webroot`** - 由Tomcat部署的应用目录名决定，这是标准的Java Web应用部署方式
2. **`/decision`** - 由应用内部的`ServerConfig`和`PlatformScaffoldBasicConfigProvider`配置决定

这种设计允许：
- 灵活部署到不同的Context Path
- 可配置的Servlet名称，避免与其他应用冲突
- 标准的Spring MVC架构，便于扩展和维护

## 七、补充：为什么叫 "decision"？

### 1. 产品定位

"decision" 这个名称来源于帆软的产品定位：

| 产品 | Servlet名称 | 含义 |
|-----|------------|------|
| FineReport | `decision` | 决策报表系统，报表用于辅助企业决策 |
| FineBI | `webroot` | BI商业智能分析系统 |

FineReport 的核心理念是"用数据驱动决策"，因此使用了 `decision` 作为核心 Servlet 的名称。

### 2. 历史演变

| 版本 | 默认Servlet名称 | 说明 |
|-----|----------------|------|
| 早期版本 | `ReportServer` | 直接以功能命名 |
| FineReport 9.0+ | `decision` | 统一决策平台概念 |
| FineReport 11.0 | `decision` | 延续至今 |

### 3. 兼容性说明

FineReport 仍保留了 `/ReportServer` 的兼容路由：

```java
// ViewRequestConstants.java:22
public static final String REPORT_VIEW_PATH_COMPATIBLE = "/view/ReportServer";
```

这意味着以下两个 URL 可能指向相同的功能：
- `/webroot/decision/view/report?xxx`
- `/webroot/decision/view/ReportServer?xxx`（兼容模式）

### 2. 拦截器与过滤器的区别

| 对比项 | 过滤器 (Filter) | 拦截器 (Interceptor) |
|-------|----------------|---------------------|
| 技术层级 | Servlet 规范 | Spring 框架 |
| 执行时机 | 在 Servlet 之前 | 在 Controller 之前/之后 |
| 配置方式 | `servletContext.addFilter()` | `InterceptorRegistry.addInterceptor()` |
| 典型用途 | 编码转换、安全头、URI检查 | 登录检查、权限验证、日志记录 |

### 3. Spring MVC 注解与路由映射

FineReport 使用 Spring MVC 的注解来定义具体的业务路由：

```java
@Controller("auth")
@LoginStatusChecker(required = false)
public class LoginResource {

    // GET /login → 返回登录页面
    @RequestMapping(value = {"/login"}, method = {RequestMethod.GET})
    public String page(HttpServletRequest req, HttpServletResponse res) {
        // ...
    }

    // POST /login → 处理登录请求
    @RequestMapping(value = {"/login"}, method = {RequestMethod.POST})
    public Response login(HttpServletRequest req, HttpServletResponse res,
                          @RequestBody LoginRequestInfoBean loginRequestInfoBean) {
        // ...
    }
}
```

完整的路由映射关系：

```
/webroot/decision/login     → LoginResource.page()
/webroot/decision/logout    → LoginResource.logout()
/webroot/decision/view/report/...  → PageRequestService
/webroot/decision/v10/user/...     → UserService
```

## 九、常见问题

### Q1: 如何修改 Servlet 名称？

通过配置修改（需要重启）：

```java
// 方式一：代码修改
ServerConfig.getInstance().setServletName("myapp");
// 修改后 URL 变为: /webroot/myapp/xxx

// 方式二：配置文件修改
// 在 FineReport 的配置数据库或配置文件中修改 servletName 属性
```

### Q2: 能否去掉 /decision 这一层？

理论上可以，但不推荐：
1. 将 Servlet 映射改为 `/*` 会导致静态资源也被拦截
2. 可能与其他 Servlet 冲突
3. 破坏原有的 URL 结构，影响插件兼容性

### Q3: /webroot 和 /decision 的区别是什么？

| 路径 | 层级 | 决定因素 | 修改方式 |
|-----|------|---------|---------|
| `/webroot` | Tomcat 层 | 应用部署目录名 | 重命名应用目录或配置 Context |
| `/decision` | 应用层 | ServerConfig 配置 | 修改 servletName 配置 |
